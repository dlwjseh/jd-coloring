# jd-coloring — 프로세스 가이드

iPad·Mac 컬러링 게임 앱을, 역할별 전문가 에이전트를 거쳐 단계적으로 만든다.
기술 스택: **SwiftUI** (iPad·Mac 멀티플랫폼)

## 4명의 전문가와 사용법

| 단계 | 에이전트 | 호출 방법 | 산출물 위치 |
|------|----------|-----------|-------------|
| 1. 기획 | 기획자 | `/기획` | `docs/01-기획/` |
| 2. 디자인 | 디자이너 | `/디자인` | `docs/02-디자인/` |
| 3. 개발 | iMac 개발자 | `/개발` | 앱 코드 (저장소 루트) |
| 4. 평가 | 개발평가자 | "평가자 서브에이전트로 검수해줘" | `docs/04-리뷰/` |

## 왜 이렇게 나눴나
- **기획·디자인·개발**은 당신과 **대화하며 구체화**해야 하므로 슬래시 커맨드(이 대화창에서 역할을 입는 방식)다.
- **평가자**는 완성된 코드를 받아 **혼자 검수하고 리포트**하면 되므로 서브에이전트(독립 실행)다.

## 핵심 원칙: 문서로 인계한다
각 단계는 **앞 단계의 문서를 읽고 → 자기 산출물을 문서로 남긴다.**
이 문서 릴레이가 전체 프로세스를 잇는 뼈대다. 그래서 복잡한 오케스트레이션 없이도 굴러간다.

```
/기획  →  01-기획/requirements.md
   ↓ (읽음)
/디자인 →  02-디자인/design-spec.md + mockups/*.svg
   ↓ (읽음)
/개발  →  앱 코드
   ↓ (읽음)
평가자  →  04-리뷰/review-*.md
```

## 추천 진행 순서
1. `/기획` 으로 시작 — 막연한 아이디어를 말하면 기획자가 질문하며 구체화한다.
2. 기획이 확정되면 `/디자인` — 목업(`02-디자인/mockups/`)을 눈으로 확인한다.
3. 디자인이 좋으면 `/개발` — 작은 단위로 나눠 구현한다.
4. 한 기능이 끝날 때마다 평가자에게 성능 검수를 맡긴다.

언제든 이전 단계로 돌아가 문서를 갱신하고 다음 단계를 다시 돌려도 된다.
